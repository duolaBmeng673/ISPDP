# 安全认证系统开发文档

## 1. 项目目标

本项目实现了一个面向课程展示的高安全性注册与登录系统，核心目标不是只完成“用户名 + 密码”的基础登录，而是在传统密码认证的基础上，引入 RSA 数字签名作为第二因子，并通过邮箱验证实现跨设备恢复，形成更完整的安全认证链路。

系统由三部分组成：

- 前端：原生 HTML、CSS、JavaScript
- 后端：Node.js + Express
- 数据库：MySQL

## 2. 总体安全设计

本项目采用四层安全设计：

### 2.1 传输层安全

- 系统默认按 HTTPS 场景设计
- 登录和注册中的敏感信息应通过 TLS 进行传输保护
- 即使在课程演示环境中使用本地 HTTP，也应在文档和答辩中明确说明，真实部署必须启用 HTTPS

### 2.2 密码存储安全

- 用户密码绝不以明文形式存入数据库
- 服务端使用 `Argon2id` 对密码进行哈希存储
- `Argon2id` 具备抗 GPU 暴力破解优势，安全性高于简单哈希，也优于无参数优化的传统方案

### 2.3 RSA 双因子认证

项目的特色在于将 RSA 引入登录流程：

1. 注册时，浏览器本地生成 RSA 密钥对
2. 公钥上传到服务端保存到数据库
3. 私钥仅保存在浏览器本地 `IndexedDB`
4. 登录时，用户先通过密码认证
5. 服务端生成随机挑战码
6. 浏览器使用本地私钥对挑战码签名
7. 服务端使用数据库中的公钥验证签名
8. 如果用户更换设备，可通过邮箱验证码完成身份恢复并绑定新的公钥

这样可以证明当前登录者同时满足两个条件：

- 知道正确密码
- 持有对应私钥

因此该系统不再是单因子密码认证，而是“密码 + 私钥持有证明”的双因子模型；同时又通过邮箱恢复机制兼顾了跨设备可用性。

### 2.4 常见攻击防护

- 防 SQL 注入：使用 `mysql2` 参数化查询
- 防会话窃取：登录态存于 `HttpOnly` Cookie
- 防暴力破解：对认证接口启用限流
- 防响应头漏洞：使用 `helmet`
- 防挑战重放：挑战码具备时效性，并在成功使用后立即标记为已使用
- 防验证码重放：邮箱验证码具备有效期，且验证成功后立即作废

## 3. 注册流程原理

注册流程如下：

1. 用户在注册页输入用户名、邮箱和密码
2. 前端调用 `Web Crypto API` 生成 2048 位 RSA 密钥对
3. 前端导出：
   - 公钥：PEM 格式，提交给后端
   - 私钥：PEM 格式，保存到本地 `IndexedDB`
4. 后端对密码使用 `Argon2id` 哈希
5. 后端将以下信息写入 MySQL：
   - 用户名
   - 邮箱
   - 密码哈希
   - RSA 公钥

设计要点：

- 私钥永不离开客户端
- 服务端只保管公钥，不具备冒充用户签名的能力

## 4. 登录流程原理

登录分为两个阶段。

### 4.1 第一阶段：密码校验

用户提交用户名和密码后：

1. 服务端根据用户名查询用户
2. 使用 `Argon2id` 验证密码是否匹配
3. 如果密码正确，服务端不立即登录，而是生成挑战码

### 4.2 第二阶段：挑战签名

服务端生成一个随机挑战码，并将其摘要存入数据库：

- 原始挑战码发送给前端
- 数据库中保存的是 `SHA-256(challenge)` 摘要，而不是挑战原文

随后前端：

1. 从 `IndexedDB` 读取本地私钥
2. 使用 RSA 私钥对挑战码进行签名
3. 将签名结果提交给后端

后端：

1. 查询该挑战记录是否存在、是否过期、是否已使用
2. 比较挑战码摘要是否一致
3. 用数据库中的公钥验证签名
4. 验签成功后才创建最终 Session

### 4.3 第三阶段：跨设备邮箱恢复

当用户更换设备，导致当前浏览器中不存在原来的私钥时，系统支持通过邮箱完成恢复：

1. 用户输入用户名和密码
2. 服务端再次确认密码正确
3. 服务端向已绑定邮箱发送 6 位验证码
4. 新设备本地生成新的 RSA 密钥对
5. 用户输入邮箱验证码
6. 服务端校验验证码后，将数据库中的公钥替换为当前设备的新公钥
7. 当前设备完成重新绑定，并自动登录

这个机制的作用是：

- 不需要把私钥同步到服务器
- 保留“私钥只在客户端保存”的安全边界
- 同时解决只能在单一设备登录的问题

## 5. 数据表设计

### 5.1 `users`

用于保存用户基本认证数据：

- `id`
- `username`
- `password_hash`
- `email`
- `email_verified`
- `public_key`
- `created_at`

其中：

- `password_hash` 保存 Argon2id 哈希
- `public_key` 保存用户的 RSA 公钥
- `email` 保存绑定邮箱
- `email_verified` 表示邮箱是否已验证

### 5.2 `login_challenges`

用于保存登录挑战信息：

- `id`
- `user_id`
- `challenge_hash`
- `expires_at`
- `used`
- `created_at`

设计特点：

- 不保存明文挑战码，只保存摘要
- 挑战码具有有效期
- 成功使用后不可再次复用

### 5.3 `email_verification_codes`

用于保存邮箱恢复验证码：

- `id`
- `user_id`
- `code_hash`
- `purpose`
- `expires_at`
- `used`
- `created_at`

设计特点：

- 不保存明文验证码，只保存哈希值
- 使用场景通过 `purpose` 区分
- 成功验证后立即失效

## 6. 关键实现原理

### 6.1 为什么不用 RSA 直接加密密码

RSA 更适合做签名和密钥交换，不适合替代密码哈希。

原因有三点：

- RSA 不适合频繁直接处理登录密码
- 即使前端把密码用 RSA 加密，服务端仍然需要验证口令本身
- 真正保护密码存储安全的关键是 `Argon2id/Bcrypt`，而不是 RSA

因此本项目把 RSA 用在最合适的位置：身份签名证明。

### 6.2 为什么私钥存本地而不是数据库

如果私钥保存在服务器或数据库中，那么服务器本身就可以伪造签名，双因子认证就失去了意义。

所以本项目规定：

- 公钥上传服务器
- 私钥只保存在客户端

这是非对称加密体系中最核心的安全边界之一。

### 6.3 为什么选择 `IndexedDB`

与 `localStorage` 相比，`IndexedDB` 更适合存储结构化敏感数据：

- 容量更大
- 异步接口更适合复杂对象处理
- 不会像普通字符串存储那样过于简陋

虽然它仍然属于浏览器侧存储，无法替代硬件安全模块，但在课程项目中已经是比 `localStorage` 更合理的选择。

### 6.4 为什么还要加入邮箱恢复

如果只采用“密码 + 本地私钥”模式，那么账号实际上会与当前浏览器环境强绑定：

- 换电脑后无法直接登录
- 清空浏览器站点数据后会丢失私钥
- 用户可用性不足

因此本项目增加了邮箱恢复机制：

- 正常登录仍以 RSA 私钥签名为主
- 只有在新设备恢复场景下，才允许通过邮箱验证码重新绑定公钥

这样既保留了高安全性的主流程，也解决了跨设备使用的问题。

## 7. 项目开发特色

本项目的主要特色如下：

- 在基础登录系统上实现了 RSA 数字签名双因子认证
- 在不上传私钥的前提下，通过邮箱验证码实现新设备恢复与公钥重绑
- 使用 MySQL 参数化查询，避免 SQL 注入
- 使用 Argon2id 存储密码，提高数据库泄露后的抗爆破能力
- 使用挑战码摘要存储，降低挑战表泄露风险
- 使用邮箱验证码哈希存储，降低邮件验证码泄露风险
- 使用 `HttpOnly` Cookie 维护会话，避免前端脚本直接读取令牌
- 使用 `helmet` 自动补充多项安全响应头
- 登录挑战具备时效性和一次性，避免重放攻击

## 8. 局限与可扩展方向

当前系统仍有可以继续扩展的空间：

- 可增加 HTTPS 证书部署，形成完整线上安全链路
- 可加入短信、邮箱或 TOTP 验证，形成更标准的多因子体系
- 可为 RSA 私钥增加导出备份与恢复机制
- 可加入设备绑定和异常登录告警
- 可进一步引入 CSRF 防护令牌
- 可增加邮箱验证码发送频率限制与图形验证码

## 9. 答辩时可直接说明的亮点

答辩时可以概括为以下几句话：

1. 我们没有把密码明文存储在数据库，而是使用 Argon2id 进行高强度哈希。
2. 我们不仅做了密码登录，还引入了 RSA 私钥签名作为第二因子。
3. 系统采用挑战应答机制，服务端只验证签名，不接触私钥。
4. 为了解决跨设备问题，我们增加了邮箱验证码恢复机制，在新设备上验证邮箱后重新绑定新的 RSA 公钥。
5. 我们使用参数化查询、限流、HttpOnly Cookie 和安全响应头共同提升整体安全性。
